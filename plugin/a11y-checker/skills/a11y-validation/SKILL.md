# Accessibility Validation

## When to use this skill

Use when you need to:

- Test web pages or components for accessibility compliance
- Validate WCAG 2.2 (Levels A, AA, AAA) conformance
- Check EN 301 549 or RaWeb standards adherence
- Perform keyboard navigation and focus management testing
- Validate ARIA implementation
- Test with browser automation (Playwright)

---

## ⚡ Architecture: Parallel Phased Assessment

> **Context overflow prevention**: Never accumulate all 264 results in working memory.  
> After completing each section, **immediately write the partial result to disk** and release it from context.  
> The final report is generated by a **separate sub-agent** that reads from disk — not from accumulated context.

### Execution Phases

```
PHASE 1 — Static DOM (Parallel Batch A + B simultaneously):
  Batch A: Document Structure · HTML Quality · ARIA · Images · Tables
  Batch B: Color/Contrast · Motion · Multimedia · Content Without CSS · Links

PHASE 2 — Interactive (Sequential Batch E then F, shared browser session):
  Batch E: Execute user flows → test interaction-relevant sections after each step
  Batch F: Keyboard Navigation · Forms · Custom Widgets · Error Prevention

PHASE 3 — Responsive/Platform (Parallel Batch C + D simultaneously):
  Batch C: Responsive/Reflow · Mobile/Touch · Consistent ID
  Batch D: Help/Documentation · PDF/Documents · Status/Feedback

PHASE 3.5 — Pre-Aggregation (orchestrator runs script, < 1 second):
  node .agents/skills/a11y-validation/scripts/aggregate-sections.js {sessionDir}
  Strips skipped checks, collects failed issues, writes summary.json

PHASE 4 — Report Generation (4 sequential sub-agents):
  Each reads summary.json only (~5K tokens) → writes one report file
```

**Each batch runs as an independent sub-agent** so context stays bounded.  
**Batches A+B** and **Batches C+D** are launched simultaneously (parallel tool calls).

## Incremental Save Pattern (MANDATORY)

After completing **each section**, write a compact partial result file — do NOT hold results in working memory.

**Save path**: `.{packagesName}/a11y-assessment/{Page}/{date}/sections/{section-key}.json`

**Compact JSON format** (minimise file size):

```json
{
  "section": "document-structure",
  "label": "Document Structure & Semantics",
  "total": 21,
  "passed": 18,
  "failed": 2,
  "skipped": 1,
  "issues": [
    {
      "id": "DS-001",
      "sev": "Critical",
      "wcag": "1.3.1",
      "check": "Heading hierarchy",
      "detail": "h3 follows h1 — h2 level missing",
      "selector": "main h3:first-child"
    }
  ]
}
```

> After saving: release the section findings from active context. Only keep the **issue list** (IDs + severity) as a running tally.

## Testing Workflow

### Phase 1 — Launch parallel static batches

Start Batch A and Batch B **simultaneously** (parallel tool calls).  
Each sub-agent:

1. Opens the page URL via playwright-extension
2. Runs its assigned sections using reference files
3. Writes one `{section-key}.json` per section to the sections folder
4. Returns a summary: `{ sections: [...], totalIssues: N }`

### Phase 2 — Interactive user flows (sequential)

For **each user flow step**, test ONLY the relevant categories:

| Interaction type       | Test sections         |
| ---------------------- | --------------------- |
| Form input/submit      | `forms`, `errors`     |
| Modal/dialog open      | `widgets`, `keyboard` |
| Navigation             | `links`, `keyboard`   |
| Dynamic content reveal | `aria`, `status`      |
| Error state trigger    | `errors`, `forms`     |

After each step: save interaction findings to `.../sections/interactive-{flow-n}.json` and clear from context.

### Phase 3 — Launch parallel platform batches

Start Batch C and Batch D simultaneously, same pattern as Phase 1.

### Phase 3.5 — Pre-Aggregation (orchestrator runs directly)

Run the aggregation script — no LLM call needed:

```bash
node .agents/skills/a11y-validation/scripts/aggregate-sections.js {sessionDir}
```

This reads all section JSONs, strips skipped checks, collects failed issues (no deduplication), sorts by severity, and writes `{sessionDir}/summary.json` in < 1 second.

### Phase 4 — Report generation (sub-agent)

Pre-requisite: `{sessionDir}/summary.json` must exist (created by Phase 3.5 script).

Launch **4 sequential** `runSubagent(agentName: "a11y-report-writer")` calls, each reading `summary.json` only:

```
Step 4a — Scorecard → a11y-report.md
Step 4b — Critical remediation → a11y-critical-remediation.md
Step 4c — Major remediation → a11y-major-remediation.md
Step 4d — Enhancement remediation → a11y-enhancement-remediation.md
```

Each sub-agent reads ~5K tokens (summary.json) instead of ~50K (all section files).

## Playwright reference

```javascript
// Keyboard navigation
await page.keyboard.press("Tab");
await page.keyboard.press("Shift+Tab");
await page.keyboard.press("Enter");
await page.keyboard.press("Space");
await page.keyboard.press("Escape");
await page.keyboard.press("ArrowDown");

// Accessibility locators
page.getByRole("button", { name: "Submit" });
page.getByLabel("Email address");
await element.getAttribute("aria-expanded");

// Computed styles
await page.evaluate(() => {
  const el = document.querySelector(".target");
  const s = window.getComputedStyle(el);
  return { color: s.color, bg: s.backgroundColor, outline: s.outline };
});
```

## Severity model

- **Critical (Level A)**: Blocks or significantly impairs core tasks
- **Major (Level AA)**: Important usability/accessibility gaps
- **Enhancement (Level AAA)**: Beneficial improvements, not mandatory

## Status taxonomy

- **Passed**: Meets success criterion with evidence
- **Failed**: Does not meet criterion; include repro steps
- **Not Tested**: Requires manual or unavailable testing
- **Not Applicable**: Criterion doesn't apply to page/component

---

## Important constraints

- **Do NOT write test code** (no .spec.ts files)
- **Do NOT mark tests as passed if not executed**
- **Do NOT invent or assume test results**
- **Mark as "Not Tested"** if checks cannot be performed
- **Save each section to disk immediately** — do not accumulate all results in context
- Create **only accessibility reports** following this format
